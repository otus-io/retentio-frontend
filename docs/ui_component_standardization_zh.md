# UI 组件统一封装需求（按钮 / 输入框 / 图标按钮）

## 背景

当前 Flutter 前端在基础交互组件上存在明显不一致：

- 按钮样式分散（`ElevatedButton`、`TextButton`、`FilledButton` 等），并且页面内重复定义样式。
- 输入框（`TextField`）高度、间距、边框与聚焦态不统一。
- 图标按钮（`IconButton`）点击区域、禁用态和视觉反馈不一致。

这会提高维护成本，也会降低产品界面的一致性。

## 目标

创建且仅创建 **3 个公共组件**，并在全项目完成替换：

1. **AppButton**
2. **AppInput**
3. **AppIconButton**

替换后要求保持现有业务行为不变，同时实现统一视觉与交互规范。

## 范围

### 包含范围

- 在共享组件层新增 3 个可复用组件。
- 替换前端页面中的按钮 / 输入框 / 图标按钮使用。
- 保持现有页面逻辑与交互（跳转、校验、回调、加载态）不变。
- 必要时更新受影响测试。

### 不包含范围

- 全量替换所有展示型 `Icon(...)`。
- 全局设计系统重构。
- 新业务功能开发。

## 现状规模（用于工时基线）

- 按钮类控件（`ElevatedButton` / `TextButton` / `FilledButton`...）：约 **12 个文件**。
- 输入框（`TextField`）：约 **5 个文件**。
- 图标按钮（`IconButton`）：约 **8 个文件**。

## 组件需求

### 1）AppButton

统一支持：

- 变体：`primary`、`secondary`、`ghost`、`danger`。
- 尺寸：`sm`、`md`、`lg`。
- 状态：可用 / 禁用 / loading。
- 可选前后图标插槽。
- 可选整行宽度（full width）。

行为要求：

- 统一最小高度与水平内边距。
- loading 态阻止重复点击。
- 各页面点击反馈与焦点反馈一致。

### 2）AppInput

统一支持：

- label / hint / helper / error 文案。
- 前后缀组件。
- 密码态（obscure）。
- 支持 controller 和回调方式。

行为要求：

- 统一高度与内容 padding。
- 统一边框圆角、默认/聚焦/错误/禁用态规则。
- 与现有表单校验流程兼容。

### 3）AppIconButton

统一支持：

- 传入 icon data 或自定义 icon widget。
- 变体：default / subtle / danger。
- 尺寸与点击热区统一。
- 需要时支持 tooltip 语义。

行为要求：

- 统一点击区域和禁用态。
- 避免局部页面点击区域过小。

## 实施计划

1. 盘点现有使用点，建立旧组件到新组件的映射。
2. 实现 3 个公共组件，优先稳定 API。
3. 按模块替换（登录注册 -> 卡组列表 -> 学习页 -> 个人页 -> 共享组件层）。
4. 运行格式化、静态检查、测试并修复回归。
5. 输出迁移说明和后续清理建议。

## 验收标准

- 业务页面中不再直接使用 `ElevatedButton`、`TextButton`、`FilledButton`、`TextField`、`IconButton`（框架特殊场景可例外并注明）。
- 登录、注册、卡组列表、学习页、个人页的按钮/输入框/图标按钮风格一致。
- `flutter analyze` 通过。
- `flutter test` 通过。
- 核心流程无明显交互回归。

## 风险与应对

- **风险：** 替换面大，容易出现视觉回归。  
  - **应对：** 按模块替换并逐模块验证。
- **风险：** 测试用例绑定旧组件类型。  
  - **应对：** 优先断言行为与语义，必要时再调整类型断言。
- **风险：** deck/video 页面存在定制控制逻辑。  
  - **应对：** 对保留旧控件的特例给出显式注释和原因。

## 预估工时（由 AI 协作实现）

- 基线范围（3组件 + 按钮/输入框/IconButton 全量替换）：**9.5 ~ 13 小时**。
- 最可能工时：**约 11 小时**。
- 若扩展到所有展示型图标 `Icon(...)`，需额外 **4 ~ 8 小时**。

## 交付物

- `AppButton`、`AppInput`、`AppIconButton` 公共组件文件。
- 当前范围内的全量替换代码。
- 受影响测试更新（如有）。
- 本需求文档（中英文）。
