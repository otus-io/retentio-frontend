## PR 提交流程

提交 PR 时，请遵循以下流程：

1. **创建 PR** - 提交你的 Pull Request 到 `main` 分支
2. **Claude 机器人审查** - 等待 Claude AI 自动审查评论
3. **队友审查** - 请求团队成员进行二次审查
4. **处理评论** - 回应 Claude 和队友的反馈意见
   - ⚠️ **不要自己关闭评论** - 让审查者在确认后关闭
5. **合并** - 获得批准并解决所有评论后，合并 PR

## 代码质量标准

### 基本原则

1. **可读性**：代码应该具有自解释性。优先使用清晰的变量/函数名，而非注释。
2. **简洁性**：保持简单，避免过度设计。
3. **可测试性**：编写易于测试的代码，为新功能编写单元测试。
4. **一致性**：遵循现有的代码模式和项目规范。

### 代码风格

- 遵循各语言的风格指南：
  - **Go**：遵循 [Effective Go](https://go.dev/doc/effective_go) 并运行 `gofmt`
  - **Kotlin/Android**：遵循 [Kotlin 编码规范](https://kotlinlang.org/docs/coding-conventions.html)
  - **Flutter/Dart**：遵循 [Effective Dart](https://dart.dev/effective-dart) 并运行 `dart format`
- 使用有意义的变量和函数名
- 保持函数职责单一
- 函数长度尽量控制在 **50 行**以内
- 行长度限制在 **120 个字符**

### 文档

- 为复杂逻辑或非显而易见的决策添加注释
- 添加新功能时更新 README 或相关文档
- 为公共 API 添加 godoc/kdoc 注释

## Pull Request 指南

### PR 标题规范

PR 标题**必须**遵循以下格式：

```
<类型>(<范围>): <主题>
```

#### 类型

| 类型 | 描述 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 仅文档更改 |
| `style` | 代码风格更改（格式化、缺少分号等） |
| `refactor` | 既不修复 bug 也不添加功能的代码更改 |
| `perf` | 性能优化 |
| `test` | 添加或更新测试 |
| `chore` | 维护任务（构建流程、依赖等） |

#### 范围

范围应指明受影响的模块或组件：
- `auth` - 认证相关更改
- `deck` - 卡组管理
- `card` - 卡片操作
- `api` - 通用 API 更改
- `frontend` - 前端/移动应用更改
- `ci` - CI/CD 流水线更改

#### 主题

- 使用祈使语气："add feature" 而非 "added feature" 或 "adds feature"
- 首字母不要大写
- 结尾不要加句号
- 保持在 **50 个字符**以内

#### 示例

```
feat(auth): add OAuth2 support for Google login
fix(deck): resolve null pointer when deck is empty
docs(api): update swagger documentation for card endpoints
refactor(card): simplify card validation logic
test(auth): add unit tests for token refresh
```

### PR 大小指南

#### 每个 PR 的代码行数

| 大小 | 更改行数 | 建议 |
|------|----------|------|
| **XS** | 1-10 | 适合快速修复 |
| **S** | 11-50 | 适合小功能或 bug 修复 |
| **M** | 51-200 | 适合中等功能 |
| **L** | 201-500 | 尽可能拆分 |
| **XL** | 500+ | **必须拆分**为更小的 PR |

**目标**：PR 的实际代码更改保持在 **200 行**以内（不包括测试和生成的文件）。

#### 为什么要小 PR？

- 更容易彻底审查
- 审查周转更快
- 引入 bug 的风险更低
- 更容易回滚
- 更好的 git 历史记录

### 每个 PR 的提交数

| 提交数 | 指南 |
|--------|------|
| **1-3** | 理想 |
| **4-5** | 较大功能可接受 |
| **6+** | 考虑拆分为多个 PR |

**目标**：PR 保持在 **3 个提交**或更少。

### PR 描述模板

每个 PR 应包含：

```markdown
## 这是什么类型的 PR？
<!-- feat / fix / docs / refactor / test / chore -->

## 描述
<!-- 详细描述您的更改 -->

## 相关 Issue
<!-- 链接到相关 issue：Fixes #123 -->

## 如何测试？
<!-- 描述您如何测试您的更改 -->

## 检查清单
- [ ] 我的代码遵循项目的代码风格
- [ ] 我已对我的代码进行了自我审查
- [ ] 我已添加测试来证明我的修复/功能有效
- [ ] 新的和现有的单元测试在本地通过
- [ ] 我已相应地更新了文档
```

## 提交规范

### 提交消息格式

```
<类型>(<范围>): <主题>

<正文>

<页脚>
```

### 提交消息规则

1. **主题行**：最多 **50 个字符**
2. **正文**：每行 **72 个字符**换行，解释*是什么*和*为什么*（而非*怎么做*）
3. **页脚**：引用 issue 和重大更改

### 原子提交

每个提交应该：
- 代表单一的逻辑更改
- 能够独立编译并通过测试
- 不破坏构建

### 每个提交的代码行数

| 行数 | 指南 |
|------|------|
| **1-50** | 理想 - 专注、易审查 |
| **51-100** | 较大功能可接受 |
| **100+** | 应拆分为多个提交 |

**目标**：提交保持在 **100 行**代码更改以内。

### 提交消息示例

**好的示例：**
```
feat(auth): add JWT token refresh endpoint

实现自动令牌刷新以改善用户体验。
用户不再需要在令牌过期时重新登录。

Closes #42
```

**不好的示例：**
```
fixed stuff
```

## PR 审核流程

### 审核要求

- 所有 PR 在合并前需要至少 **1 个批准**
- CI 检查必须通过
- 没有未解决的对话

### 审核者职责

1. 尽可能在 **24-48 小时**内审核代码
2. 提供建设性反馈
3. 只有在对更改有信心时才批准
4. 使用 GitHub 的建议功能进行小修复

### 审核标签

| 标签 | 描述 |
|------|------|
| `lgtm` | Looks Good To Me - 已批准 |
| `needs-rebase` | PR 需要在 main 上变基 |
| `needs-tests` | PR 需要额外的测试 |
| `needs-docs` | PR 需要文档更新 |
| `do-not-merge` | PR 暂时不应合并 |

### 合并策略

- 功能分支使用 **Squash and Merge**
- 确保压缩后的提交消息遵循规范
- 合并后删除分支
