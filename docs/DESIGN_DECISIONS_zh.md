# 设计决策

本文档记录了 WordUpX 项目中的关键设计决策及其背后的原因。

---

## 为什么选择 RESTful API 而非统一/RPC 端点

在设计 API 时，我们评估了两种方案：**标准 RESTful URL**（如 `GET /decks/5`、`DELETE /cards/2`）和**统一/RPC 端点**（将 ID 和操作都放在请求体中，通过单一端点如 `POST /api/endpoint` 处理所有请求）。

我们最终选择了 RESTful 风格，原因如下——统一/RPC 端点存在以下潜在问题：

### 1. 零缓存（性能损失）

浏览器和 CDN 依赖 **URL** 和 **GET 请求**来缓存数据。由于 RPC 风格对所有操作都使用 `POST` 请求，浏览器无法缓存响应结果。这意味着每次用户浏览页面时，服务器都必须重新处理完全相同的请求，造成不必要的性能开销。

### 2. "黑盒"日志（可观测性差）

服务器访问日志会变得毫无用处。与其看到清晰明了的 `GET /decks/5` 或 `DELETE /cards/2`，每条日志都会变成千篇一律的 `POST /api/endpoint`。在这种情况下，你无法轻松地排查 Bug、追踪流量模式或了解哪些资源最受欢迎——除非额外构建复杂的自定义日志系统。

### 3. 安全盲区

Web 应用防火墙（WAF）针对 **URL 扫描**做了大量优化，能够自动检测恶意模式（如 SQL 注入）。如果将 ID 和操作指令深藏在 JSON 请求体中，标准安全工具可能会遗漏那些本应被自动拦截的攻击。

### 4. 错误处理机制失效

在 RPC 模式下，HTTP 状态码通常总是返回 `200 OK`（仅表示消息到达了服务器），即使操作实际上失败了（例如"找不到该卡片"）。这会破坏依赖标准状态码的监控工具——它们依靠 `404`（未找到）、`401`（未授权）等状态码来发出告警。RESTful API 则能正确利用 HTTP 状态码，使监控和错误追踪变得直观且标准化。
